# Implementation Plan — REQ-031

## Approach

Build a single shared `<CustomizationPicker>` component that renders a `MenuItem`'s `customizations[]` (groups → options) using radios for required groups and checkboxes for optional groups. Mount it in three order-creation surfaces: the customer detail modal, the admin Edit Order dialog, and the staff Express Order create page. Plumb the selected `{ name, option, price }[]` triples through cart-store → checkout → public order POST (already accepted), through `expressCreateOrderAction`, and through `updateOrderItemsAction`. Add defensive server-side validation in those three actions so submitted `(group, option)` pairs must exist on the menu item before the order is created. Ship a user-facing PDF guide alongside the code so the round-trip can be exercised without engineering help.

## Architecture Decisions

- **D1 — Single shared picker component.** Three order surfaces, one component. Eliminates the risk of behavioural drift between customer and admin paths. Mirrors REQ-029's single-source `lib/expense-search.ts` pattern.
- **D2 — Required vs optional group rendering.** `group.required === true` → RadioGroup (one selection mandatory). `group.required === false` → independent Checkboxes (zero or many). Picker exposes `isValid` for hosts to disable submit on invalid.
- **D3 — Cart merge key includes customizations.** Two orders of Poundo with different soups must be two cart lines. Hash key = `${menuItemId}|${portionSize}|${JSON.stringify(customizations.sort)}`. Items with no customizations keep existing merge behaviour. Bump `persist` version so legacy localStorage carts don't mis-merge after deploy.
- **D4 — Selected customizations stored as `{ name, option, price }` triples.** Matches `IOrderItem.customizations` shape and the public POST API contract. No new wire format.
- **D5 — Server-side validation is mandatory in every order-creating action.** Defence in depth: a tampered or stale client must not be able to inject `(groupName, optionName)` pairs that don't exist on the menu item. Single-source via `lib/customization-validation.ts` so the rule is identical across the three actions.
- **D6 — Customization surcharges ARE folded into line subtotals in this REQ.** The venue uses non-zero option prices (e.g. premium-soup options). Math: `lineTotal = (basePrice + Σ option.price) × quantity × portionMultiplier`. Surcharge scales with portion (Option B) — half-Poundo deducts half-Egusi (REQ-030) and bills half-Egusi (this REQ). Legacy items with no customizations or zero surcharges are unchanged (Σ = 0).
- **D7 — Linked-inventory pre-validation NOT in this REQ.** REQ-030 silently skips missing linked inventories at fulfilment time; we keep that policy. `validateCartItem` does not reach for linked stock. Tracked as follow-up issue [#68](https://github.com/metasession-dev/wawagardenbar-app/issues/68).
- **D8 — User-docs PDF is part of the deliverable** (gates AC10). Same template as `docs/locations-inventory-guide.html`, references REQ-030 + REQ-031 + release tickets so end users see how to set the feature up and use it at order time.
- **D9 — Admin Edit Order picker uses a sub-dialog**, not inline rows. Existing edit dialog is already dense; modal-on-modal (Radix Dialog inside Dialog supported) keeps layout clean.
- **D10 — Admin builder shows a live "Combined price" preview alongside the surcharge input.** Same data model (surcharge stored), friendlier admin UX. Admin enters either field; the other auto-derives. `Surcharge: ₦500   →   {ItemName} + {OptionName} = ₦{base + 500}`. Multi-group case: surcharges sum naturally. Set surcharge to 0 if a particular option should not change the combined price.

## Dependencies

None. All UI primitives (`RadioGroup`, `Checkbox`, `Dialog`, `Label`) already in `components/ui/`.

## Risks / Considerations

- **Cart-store persist version bump** drops in-flight customer carts on first load post-deploy. Acceptable — the alternative (silent merge of legacy items into new schema) risks losing soup choices. Document in release ticket so support knows to expect "my cart is empty" reports for ~24h.
- **Modal-on-modal accessibility** in Edit Order dialog. Sub-dialog inside Edit Order needs Playwright check on focus-trap return-to-parent. Tested in the e2e spec.
- **Express Order page is large** (491 lines). Picker integration adds state but the page already manages an `items` array — extension follows existing pattern.
- **Public POST endpoint accepted `customizations` without validation pre-REQ-031.** This REQ closes a real (small) input-validation hole that REQ-030 didn't notice. Worth calling out in `ai-prompts.md` and the security summary.
- **Two human reviewers required** (HIGH risk per AI involvement +1). Plan PR for `ostendo-io` + one second reviewer.
- **Pricing-math change touches every order, not just customization orders.** Line-total formula is unified to `(base + Σ surcharge) × qty × portionMultiplier`. Legacy items pass through with `Σ = 0` and reduce to today's behaviour. AC8 (legacy-safe) is the regression gate; the full Playwright cart/checkout/order suite must show no monetary deltas for orders without customizations.
- **Server-side total reconciliation**: client-computed totals must be re-derived server-side. If the difference exceeds a 1-naira rounding tolerance, reject the request — protects against a tampered client paying less than the menu defines.

## Post-Deploy Actions

- **None at the database level** — no schema changes. All three actions remain backward-compatible (customizations array is optional everywhere).
- **Cart persist version bump** is a code change only — handled automatically by zustand on first load.
- **Re-publish `docs/customization-options-user-guide.pdf`** to whatever doc-host the team uses (or attach to the release ticket).
