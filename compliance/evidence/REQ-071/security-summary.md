# REQ-071 — Security summary

## Surface review

This REQ adds **test code only** — zero changes to production application code. The route handlers and API key validation logic are unchanged.

## STRIDE pass on the new test infrastructure

| Threat                     | Surface                                                                    | Status                                                                                                                                                                                                                         |
| -------------------------- | -------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Spoofing**               | Test creates an ephemeral API key with read scopes                         | Key is generated by the production `ApiKeyService.createKey` flow (bcrypt-hashed, prefixed `wawa_`, unique). Plaintext captured in spec-local scope; never logged, never written to env/file. Revoked + deleted in `afterAll`. |
| **Tampering**              | Test GETs against UAT public endpoints                                     | Read-only scopes (`menu:read`, `inventory:read`, `orders:read`, etc.). Cannot mutate state.                                                                                                                                    |
| **Repudiation**            | Production `lastUsedAt` timestamp is updated on each call                  | Audit trail preserved; the ephemeral key appears in `apikeys` with `name: 'e2e-req071-*'` until revoked + deleted at `afterAll`.                                                                                               |
| **Information disclosure** | Test reads back the response payload to assert shape                       | Response data is the same payload a real API consumer would see with a valid key. No PII enumerated; test only checks structural shape (e.g. `typeof item.name === 'string'`, not the value).                                  |
| **Denial of service**      | 8 sequential GET requests per run (rate-limit-respecting)                  | Created key has `rateLimit: 300` rpm; specs configured `mode: 'serial'`. Negligible UAT load.                                                                                                                                  |
| **Elevation of privilege** | Test invokes `ApiKeyService.createKey` directly from the Playwright runner | Same code path as production `POST /api/admin/api-keys` (gated behind super-admin auth there). The spec's elevation is bounded to the createdBy = SYSTEM_USER_ID; key is revoked + deleted before the run ends.                |

## Secret handling

- The ephemeral API key's plaintext is captured in a JS variable scope: `let apiKey: string;` declared above the describe block.
- Never logged, never written to env vars, never persisted in test artifacts.
- Plaintext can only be reconstructed by running the test — production storage holds only the bcrypt-hashed value.
- If the spec crashes mid-run leaving an active key, manual cleanup via `apikeys` filter on `name: /^e2e-req071-/` removes it.

## What this REQ does NOT change

- Production route handlers: unchanged.
- Production API key validation logic: unchanged.
- Production rate limits: unchanged (the spec's key has `rateLimit: 300` rpm but the rate-limit middleware itself is unchanged).

## Compliance posture

- **No new auth surface.** Spec runs server-side in the Playwright runner.
- **No new data egress.** Read endpoints only.
- **No new packages.** Uses `@playwright/test` + Node `fetch` (already available).
- **Read-only.** No production data mutated.
