Customization Options — User Guide

Wawa Garden Bar · Setup, sale, and verification · 2026-04-27

This guide explains how to set up customization options on a menu item — for example, a Poundo dish that comes with a choice of soup (Ogbono, Egusi, Ugu) — and how that selection drives multi-component inventory deduction at sale time. Each soup is its own inventory record; selling Poundo+Egusi decrements both the Poundo stock and the Egusi stock, with two distinct stock-movement audit rows.

1. What customization options are (and aren't)

A customization group is a set of choices a customer makes when ordering a parent dish. Two flavours:

Group typeBehaviourExample
RequiredCustomer must pick exactly one option (radio buttons). Order cannot be placed until selected."Soup" group on Poundo — must pick one
OptionalCustomer may pick zero or more (checkboxes)."Extras" — Plantain, Sauce, etc.

Each option can carry an inventory link. When the customer selects an option that has a linked inventory record, that record decrements alongside the parent dish's inventory at sale time.

Two REQs together This feature is the combined work of REQ-030 (the inventory-link plumbing — already on develop) and REQ-031 (the order-time picker, surcharge math, and end-to-end UX — this release). The original release ticket REQ-030 was superseded; the combined story ships under REQ-031 / issue #67.

2. Pre-requisite — set up the soup/garnish as its own menu item with inventory

The "Deduct from inventory" dropdown in the customization builder is populated from existing Inventory records, each of which is owned by a parent menu item. So before you can link "Poundo's Egusi option" to anything, Egusi itself has to exist as a menu item with inventory tracking turned on.

  1. Sign in as super-admin. Navigate to /dashboard/menu.
  2. Click Add Menu Item. Name it Egusi. Save it.
  3. Toggle Track Inventory on. Set the initial stock and unit (e.g. 10 portions). Save.
  4. Repeat for each soup / garnish you want to offer (Ogbono, Ugu, etc.).

If you already have these as standalone menu items with inventory, skip this step.

3. Configuring the customization group on the parent dish

  1. Open the parent (e.g. Poundo) at /dashboard/menu → click Edit on the row.
  2. Scroll to Customization Options.
  3. Click Add Group. Name it Soup.
  4. Toggle Required ON if the customer must pick one (recommended for soup).
  5. For each soup variant, click Add Option and enter the option name (Ogbono, Egusi, Ugu).

4. Setting per-option pricing — surcharge or combined price

Each option has a price field. This represents the surcharge added when the customer picks the option. To make the admin UX clearer, the page shows a live preview of the resulting combined price:

You enter (Surcharge)Live preview shows
0Poundo + Ogbono = ₦2,000
500Poundo + Egusi = ₦2,500
800Poundo + Ugu = ₦2,800

If you'd rather think in terms of the combined price, expand Set combined price instead below the surcharge field and type the total directly — the surcharge auto-derives.

Half & quarter portions scale the surcharge If the customer orders a half Poundo with Egusi at ₦500 surcharge, they pay 0.5 × (2,000 + 500) = ₦1,250 and the kitchen deducts half a portion of Egusi from inventory. The surcharge scales with the portion multiplier.

5. Linking each option to its inventory record

  1. For each option, find the Deduct from inventory (optional) dropdown beneath the surcharge field.
  2. Select the matching inventory item (e.g. pick Egusi for the Egusi option).
  3. The Units to deduct (portions) input appears next to the dropdown. Default 1 means one ordered Poundo+Egusi = one Egusi portion deducted. Change to 0.5 if a Poundo only consumes half a soup, etc.
  4. Repeat for every option that should deduct extra inventory. Leave the dropdown as — None — for any option that shouldn't deduct (e.g. "no soup").
  5. Click Save Changes at the top of the page.

6. Selling: customer flow on /menu

  1. The customer signs in and opens /menu.
  2. They tap the parent dish (Poundo). The detail modal opens with the Soup picker between Portion Size and Quantity.
  3. The Soup picker renders as radio buttons (because the group is required). Each option shows the surcharge if non-zero (Egusi +₦500).
  4. The Add to Cart button is disabled until the customer picks a soup. A red "please choose one" hint appears next to the group label.
  5. After selecting Egusi, the modal's total updates to ₦2,500. The customer sets quantity, optionally adds Special Instructions, and clicks Add to Cart.
  6. The cart sidebar shows the line: Poundo, with Soup: Egusi beneath it, ₦2,500 each, and the surcharge-aware line total.
  7. Checkout completes the order. The persisted order line records the customization for fulfilment.

7. Selling: staff flow on Express Order

  1. Staff opens /dashboard/orders/express/create-order.
  2. Click on the parent dish (Poundo) in the menu grid.
  3. If the dish has customization groups, a Customize Poundo dialog opens — the same picker as the customer flow.
  4. Pick the soup. Add to Order enables once required groups are satisfied.
  5. The cart summary on the right shows the line with Soup: Egusi beneath it and the surcharge-inclusive price.
  6. Two Egusi orders + one Ogbono order create three separate cart lines (different customizations = different lines).
  7. Click Checkout to either add to a tab or pay now.

8. Editing an existing order's customizations

  1. Open the order at /dashboard/orders/{orderId}.
  2. Click Edit Order Items (visible only on unpaid, non-completed, non-cancelled orders).
  3. To add a new line, pick a menu item from the dropdown. If the item has customizations, the picker sub-dialog opens automatically.
  4. Make selections, click Add to Order. The new line appears with the customizations summary beneath the item name.
  5. Quantity controls work as before. Subtotal updates surcharge-aware.
  6. Click Save Changes to persist.
You can't edit existing line customizations directly The Edit Order dialog can add new lines or change quantities. To change a line's customization (Egusi → Ogbono), remove the line and add a new one.

9. What happens at fulfilment — multi-component deduction

When staff marks the order Complete, the system runs deductStockForOrder:

  • The parent dish's inventory decrements by quantity × portionMultiplier.
  • For each customization option with an inventory link, the linked inventory decrements by quantity × portionMultiplier × inventoryDeduction.
  • One stock-movement row is written per inventory — both with category sale, both linked to the order ID for traceability.

So a sale of 2 × Poundo + Egusi writes two movement rows: −2 Poundo and −2 Egusi. A sale of 1 × half-Poundo + Egusi writes −0.5 Poundo and −0.5 Egusi.

10. What happens on cancellation

Cancelling an already-fulfilled order calls restoreStockForOrder: both the parent and linked inventories are restored symmetrically with new addition stock-movement rows tied to the same order ID. Cancellation of a not-yet-completed order does nothing (no stock had been deducted).

12. Common pitfalls & FAQ

Q. The picker shows but Add to Cart is greyed out.

You haven't picked an option in a Required group. Look for the red "please choose one" hint next to the group label.

Q. The "Deduct from inventory" dropdown is empty when I'm configuring an option.

You haven't set up any inventory records yet, OR the menu item dropdown only shows items where Track Inventory is on. Go back to step 2.

Q. I deleted the linked inventory record. What happens to existing orders?

The order-fulfilment path silently skips the linked deduction (no crash) — the parent inventory still decrements normally. New orders that try to pick that option will see the option grey-out as orphaned. Re-link it or remove it.

Q. Two customers ordered Poundo+Egusi but they merged into one cart line. Why?

Same item + same portion + same customizations + same special instructions = same cart line. That's the merge rule. Different customizations create separate lines automatically.

Q. The price the customer paid doesn't match what I see in the menu math.

The server recomputes the subtotal from menu data on submit; the customer's total is always the menu's price + surcharge × quantity × portion multiplier, plus tax/service-fee/delivery. If the displayed total differs by more than 1 naira (rounding), the server rejects the submission — so you should never see this in practice.

Q. Can the surcharge be negative (a discount for a particular option)?

Not currently. The surcharge field is treated as ≥ 0 by the math. If you need an option to be cheaper than the parent base, model it with the option as the parent and the cheaper variant as the base. We may revisit if real demand emerges.

Q. Why isn't the customer charged separately for each linked inventory item?

The line price is (parent base + Σ option surcharge) × quantity × portionMultiplier. Linked-inventory deduction is an audit/restock concern; pricing is set by the option's surcharge, not by the linked inventory's standalone price.
